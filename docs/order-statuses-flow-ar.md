## نظرة عامة

نظام الطلبات يدعم وضعين للعمل:
- **القبول التلقائي** (`auto_approve_orders = true`): الطلبات تقبل تلقائياً
- **القبول اليدوي** (`auto_approve_orders = false`): البائع يقبل الطلبات يدوياً

---

## حالات الطلب

### 1. 🆕 pending (بانتظار القبول)
**الوصف**: الطلب تم استلامه وينتظر قبول البائع

**متى تحدث**:
- عند إنشاء أي طلب جديد (في كلا الوضعين: التلقائي واليدوي)

**من يراها**:
- ✅ البائع (Vendor): يجب أن يقبل أو يرفض
- ✅ السائق (في الوضع التلقائي): يرى الطلب لكن **لا يمكنه قبوله** حتى يقبله البائع
- ❌ السائق (في الوضع اليدوي): لا يرى الطلب بعد
- ✅ العميل: يرى أن الطلب قيد المعالجة

**الإجراءات المتاحة**:
- البائع: قبول (→ processing) أو رفض (→ cancelled)
- السائق: **مشاهدة فقط** في الوضع التلقائي - لا يمكن القبول
- العميل: إلغاء الطلب (→ cancelled)

---

### 2. 🔄 processing (قيد التجهيز)
**الوصف**: الطلب مقبول من البائع وجاهز للتوصيل

**متى تحدث**:
- بعد أن يقبل البائع الطلب (في كلا الوضعين)

**من يراها**:
- ✅ البائع: يرى أن الطلب مقبول
- ✅ السائق: **يمكنه الآن قبول الطلب والبدء في التوصيل**
- ✅ العميل: يرى أن الطلب قيد التجهيز

**الإجراءات المتاحة**:
- السائق: قبول الطلب (→ driver_assigned)
- البائع: إلغاء الطلب (→ cancelled)
- العميل: إلغاء الطلب (→ cancelled)

**ملاحظة مهمة**:
- السائقون يمكنهم قبول طلبات بحالة `processing` فقط
- في الوضع التلقائي: السائقون كانوا يرون الطلب من قبل (pending) ولكن الآن يمكنهم قبوله
- في الوضع اليدوي: الطلب يظهر للسائقين لأول مرة
- يمكن لأكثر من سائق رؤية نفس الطلب
- أول سائق يقبل يحصل على الطلب

---

### 3. 👤 driver_assigned (تم تعيين سائق)
**الوصف**: سائق قبل الطلب وفي طريقه لاستلامه

**متى تحدث**:
- عندما يقبل سائق طلباً بحالة `processing`

**من يراها**:
- ✅ البائع: يرى معلومات السائق
- ✅ السائق المعين: يرى تفاصيل الطلب والموقع
- ❌ السائقون الآخرون: لا يرون الطلب بعد الآن
- ✅ العميل: يرى معلومات السائق

**الإجراءات المتاحة**:
- السائق: استلام الطلب من البائع (→ picked_up)
- السائق: إلغاء القبول (→ processing)
- البائع: إلغاء الطلب (→ cancelled)

**البيانات المحفوظة**:
```json
{
  "driver_id": "uuid-of-driver",
  "driver_name": "اسم السائق",
  "driver_phone": "+966xxxxxxxxx",
  "driver_assigned_at": "2024-02-02T10:30:00Z"
}
```

---

### 4. 📦 picked_up (تم الاستلام من البائع)
**الوصف**: السائق استلم الطلب من البائع وفي طريقه للعميل

**متى تحدث**:
- عندما يؤكد السائق استلام الطلب من البائع

**من يراها**:
- ✅ البائع: يرى أن الطلب خرج للتوصيل
- ✅ السائق: يرى موقع العميل والمسار
- ✅ العميل: يرى أن السائق في الطريق + تتبع GPS

**الإجراءات المتاحة**:
- السائق: تأكيد التسليم للعميل (→ delivered)
- السائق: الإبلاغ عن مشكلة (→ issue)

**الميزات النشطة**:
- 📍 تتبع موقع السائق GPS
- 📞 إمكانية الاتصال بين السائق والعميل
- 🗺️ عرض المسار على الخريطة
- ⏱️ وقت الوصول المتوقع

---

### 5. ✅ delivered (تم التوصيل)
**الوصف**: الطلب تم توصيله بنجاح للعميل

**متى تحدث**:
- عندما يؤكد السائق تسليم الطلب للعميل

**من يراها**:
- ✅ الجميع (سجل نهائي)

**الإجراءات المتاحة**:
- العميل: تقييم الخدمة ⭐
- لا يمكن تعديل الطلب بعد هذه الحالة

**البيانات المحفوظة**:
```json
{
  "delivered_at": "2024-02-02T11:00:00Z",
  "delivery_duration_minutes": 30,
  "total_trip_time_minutes": 45
}
```

---

### 6. ❌ cancelled (ملغي)
**الوصف**: الطلب تم إلغاؤه

**متى تحدث**:
- البائع رفض الطلب (في وضع القبول اليدوي)
- البائع ألغى الطلب لأي سبب
- السائق ألغى بعد القبول (رجوع إلى processing)
- العميل ألغى الطلب
- مشكلة في التوصيل (issue)

**من يمكنه الإلغاء**:
| الحالة | البائع | السائق | العميل |
|--------|--------|---------|---------|
| pending | ✅ | ❌ | ✅ |
| processing | ✅ | ❌ | ✅ |
| driver_assigned | ✅ | ✅* | ⚠️** |
| picked_up | ⚠️*** | ✅ | ⚠️**** |
| delivered | ❌ | ❌ | ❌ |

*السائق: يلغي القبول ويرجع الطلب لـ processing
**العميل: يحتاج تواصل مع السائق
***البائع: فقط في حالات خاصة
****العميل: يحتاج تواصل مع الدعم

**البيانات المحفوظة**:
```json
{
  "cancelled_at": "2024-02-02T10:45:00Z",
  "cancelled_by": "vendor/driver/customer",
  "cancellation_reason": "السبب"
}
```

---

### 7. ⚠️ issue (مشكلة في التوصيل)
**الوصف**: حدثت مشكلة أثناء التوصيل

**متى تحدث**:
- السائق لا يجد العميل
- العنوان خاطئ
- العميل رفض الاستلام
- مشكلة في الطلب نفسه

**الإجراءات**:
- يتواصل الدعم مع جميع الأطراف
- يمكن حل المشكلة (→ picked_up)
- أو إلغاء الطلب (→ cancelled)

## الأطراف المعنية

### 🏪 البائع (Vendor)
**صلاحياته**:
- قبول/رفض الطلبات (في وضع القبول اليدوي)
- رؤية جميع الطلبات الخاصة به
- إلغاء الطلبات قبل التوصيل
- تفعيل/تعطيل القبول التلقائي

**ما يراه**:
```sql
SELECT * FROM orders
WHERE vendor_id = 'vendor-uuid'
AND status IN ('pending', 'processing', 'driver_assigned', 'picked_up')
ORDER BY created_at DESC
```

---

### 🚗 السائق (Driver)
**صلاحياته**:
- رؤية الطلبات المتاحة (processing فقط)
- قبول الطلبات
- إلغاء القبول (قبل الاستلام)
- تأكيد الاستلام والتسليم
- الإبلاغ عن مشاكل

**ما يراه**:
```sql
-- الطلبات المتاحة
SELECT * FROM orders
WHERE status = 'processing'
AND driver_id IS NULL
ORDER BY created_at ASC

-- طلباتي الحالية
SELECT * FROM orders
WHERE driver_id = 'driver-uuid'
AND status IN ('driver_assigned', 'picked_up')
```

---

### 👤 العميل (Customer)
**صلاحياته**:
- إنشاء طلبات جديدة
- إلغاء الطلب (قبل picked_up)
- تتبع الطلب
- تقييم الخدمة

**ما يراه**:
```sql
SELECT * FROM orders
WHERE customer_id = 'customer-uuid'
ORDER BY created_at DESC
```

## سيناريو المعالجة التلقائية

### ⚡ auto_approve_orders = true

```
📱 العميل يطلب
    ↓
🆕 يتم إنشاء الطلب
    ↓
⏳ status = 'pending'
    ↓
🔔 إشعار للبائع: "طلب جديد يحتاج موافقة"
🔔 إشعار للسائقين: "طلب جديد متاح قريباً!"
    ↓
👀 البائع والسائقون يرون الطلب
    ↓
    ┌─────────┴─────────┐
    ↓                   ↓
  ✅ البائع يقبل    ❌ البائع يرفض
    ↓                   ↓
processing          cancelled
    ↓                   └→ 🔔 "تم إلغاء الطلب"
✋ السائقون يمكنهم قبول الطلب الآن
    ↓
👤 سائق يقبل: status = 'driver_assigned'
    ↓
🔔 إشعار للبائع: "السائق [اسم] في الطريق"
🔔 إشعار للعميل: "السائق في الطريق"
    ↓
🏪 السائق يصل للبائع
    ↓
📦 السائق يستلم: status = 'picked_up'
    ↓
🔔 إشعار للعميل: "الطلب في الطريق إليك"
    ↓
🏠 السائق يصل للعميل
    ↓
✅ تسليم: status = 'delivered'
    ↓
🔔 إشعار للجميع: "تم التوصيل بنجاح"
    ↓
⭐ العميل يقيّم الخدمة
```

**الفرق عن القبول اليدوي**: السائقون يرون الطلب من البداية (pending) لكن لا يمكنهم قبوله حتى يوافق البائع

**مدة الرحلة المتوقعة**: 25-50 دقيقة

## الفرق الجوهري بين النظامين

### 🔍 الفكرة الأساسية

**المعالجة التلقائية** (`auto_approve_orders = true`):
- السائقون يرون الطلب **فوراً** عند إنشائه (حالة pending)
- لكنهم **لا يمكنهم قبوله** حتى يوافق البائع
- الهدف: السائقون جاهزون ومستعدون مسبقاً
- النتيجة: توصيل أسرع بعد موافقة البائع

**القبول اليدوي** (`auto_approve_orders = false`):
- السائقون **لا يرون** الطلب حتى يقبله البائع
- بعد قبول البائع يظهر الطلب للسائقين لأول مرة
- الهدف: رقابة كاملة على متى يتم إظهار الطلب
- النتيجة: عملية أبطأ لكن أكثر تحكماً

### 📊 مثال عملي

**طلب جديد في الساعة 10:00 صباحاً**

#### المعالجة التلقائية:
```
10:00 - طلب جديد → pending
        📱 البائع يرى الطلب
        📱 السائقون يرون الطلب (لكن زر القبول معطل)

10:02 - البائع يقبل → processing
        ✅ السائقون يمكنهم الآن القبول

10:03 - سائق يقبل → driver_assigned
        (التأخير دقيقة واحدة فقط بعد موافقة البائع!)
```

#### القبول اليدوي:
```
10:00 - طلب جديد → pending
        📱 البائع فقط يرى الطلب
        ❌ السائقون لا يرون شيء

10:02 - البائع يقبل → processing
        ✅ الطلب يظهر للسائقين لأول مرة

10:05 - سائق يقبل → driver_assigned
        (التأخير 3 دقائق بعد موافقة البائع)
```

## مقارنة بين النظامين

| الميزة | المعالجة التلقائية ⚡ | القبول اليدوي 👤 |
|--------|-------------------|------------------|
| **الحالة الأولى** | pending | pending |
| **رؤية السائقين** | فورية (pending) | بعد قبول البائع (processing) |
| **قبول السائقين** | بعد موافقة البائع | بعد موافقة البائع |
| **تدخل البائع** | **إجباري** - يجب القبول/الرفض | إجباري - يجب القبول/الرفض |
| **الفرق الرئيسي** | السائقون يرون الطلب مبكراً | السائقون ينتظرون قبول البائع |
| **ميزة للسائقين** | جاهزية مسبقة للطلب | لا توجد |
| **سرعة التوصيل** | ⭐⭐⭐⭐ أسرع | ⭐⭐⭐ عادي |
| **تجربة العميل** | ⭐⭐⭐⭐ جيدة | ⭐⭐⭐ عادية |
| **سيطرة البائع** | ⭐⭐⭐⭐⭐ كاملة | ⭐⭐⭐⭐⭐ كاملة |
| **مناسب لـ** | محلات تريد سرعة + رقابة | محلات تحتاج رقابة كاملة |

## الاستعلامات المفيدة

### البحث عن الطلبات المتاحة للسائقين
```sql
SELECT
  o.*,
  v.name as vendor_name,
  v.address as pickup_address
FROM orders o
JOIN vendors v ON o.vendor_id = v.id
WHERE o.status = 'processing'
  AND o.driver_id IS NULL
ORDER BY o.created_at ASC
```

### طلبات البائع التي تحتاج موافقة
```sql
SELECT * FROM orders
WHERE vendor_id = 'vendor-uuid'
  AND status = 'pending'
ORDER BY created_at ASC
```

### الطلبات النشطة للسائق
```sql
SELECT * FROM orders
WHERE driver_id = 'driver-uuid'
  AND status IN ('driver_assigned', 'picked_up')
ORDER BY created_at ASC
```

### تتبع طلب معين
```sql
SELECT
  o.*,
  v.name as vendor_name,
  d.name as driver_name,
  d.phone as driver_phone
FROM orders o
LEFT JOIN vendors v ON o.vendor_id = v.id
LEFT JOIN drivers d ON o.driver_id = d.id
WHERE o.id = 'order-uuid'
```

## ملاحظات مهمة

### ⚠️ نقاط حرجة

1. **عدم فقدان الطلبات**:
   - الطلب يجب أن يبقى بحالة `processing` حتى يقبله سائق
   - لا يتم حذف الطلبات أبداً

2. **تعارض السائقين**:
   - إذا قبل سائقان نفس الطلب في نفس الوقت
   - الأول في قاعدة البيانات يفوز
   - الثاني يحصل على خطأ

3. **الإشعارات الفورية**:
   - استخدام Realtime Subscriptions
   - Push Notifications للخلفية
   - Sound Alerts للطلبات الجديدة

4. **تتبع GPS**:
   - يتم تفعيله فقط في حالة `picked_up`
   - يتم إيقافه بعد `delivered`
   - حفظ البطارية

## التكامل مع الكود

### مثال: سائق يقبل طلب

```typescript
async function acceptOrder(orderId: string, driverId: string) {
  const { data, error } = await supabase
    .from('orders')
    .update({
      status: 'driver_assigned',
      driver_id: driverId,
      driver_assigned_at: new Date().toISOString()
    })
    .eq('id', orderId)
    .eq('status', 'processing') // شرط مهم!
    .is('driver_id', null)      // شرط مهم!
    .select()
    .single();

  if (error) {
    throw new Error('الطلب غير متاح أو تم قبوله من سائق آخر');
  }

  return data;
}
```

### مثال: الاشتراك في الطلبات الجديدة

```typescript
// للسائقين
supabase
  .channel('new-orders')
  .on('postgres_changes', {
    event: 'UPDATE',
    schema: 'public',
    table: 'orders',
    filter: 'status=eq.processing'
  }, (payload) => {
    playSound('notification.wav');
    showNotification('طلب جديد متاح!');
  })
  .subscribe();
```

## الخلاصة

### المعالجة التلقائية: ✅ مناسب لـ
- محلات تريد توصيل سريع
- عندما يكون هناك سائقون كثر
- البائع يريد رقابة لكن يريد سرعة أيضاً
- تركيز على: **رقابة + سرعة معاً**

### القبول اليدوي: ✅ مناسب لـ
- محلات تريد رقابة كاملة
- عندما يكون الطلب يحتاج تحضير أولاً
- البائع يريد تأكيد التوفر قبل إظهار الطلب
- تركيز على: **الرقابة الكاملة**

### 💡 الخلاصة
**كلا النظامين** يعطي البائع حق القبول/الرفض، لكن الفرق في **متى يرى السائقون الطلب**
